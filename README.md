# Kill Team Solo Tracker — экспорт логики (без дизайна)

Три файла в этой папке вместе описывают ВСЮ функциональность трекера,
без единого дизайнерского решения из предыдущей реализации. Цель —
дать чистую отправную точку для сборки интерфейса с нуля.

## Что внутри

### 1. `FUNCTIONAL_SPEC.md`
Человекочитаемое описание того, что приложение должно делать: экраны,
данные, действия пользователя, правила целостности. Читать первым —
это контракт функциональности. Специально никак не намекает на то,
как это должно выглядеть.

### 2. `game_data.json`
Фактические игровые данные (правила Kill Team), которые в старой
реализации были зашиты в код: 9 позиций Crit Op, таблица Kill Op,
Tac Op по архетипам, универсальное снаряжение, снаряжение фракций,
описание двух команд (Plague Marines, Celestian Insidiants) с
операторами и их характеристиками. Чистый JSON, без разметки.

Структура верхнего уровня:
```
{
  "critOps": [...],
  "killOp": { "name", "desc", "table": { "headers", "rows" } },
  "tacOpsByArchetype": { "Recon": [...], "Infiltration": [...], ... },
  "universalEquipment": [...],
  "equipmentSelectionLimit": 4,
  "factionEquipmentByTeam": { "Plague Marines": [...], ... },
  "killTeams": { "Plague Marines": {...}, "Celestian Insidiants": {...} }
}
```

Это тот же набор данных, что в бывшем `teams.json` + константы, которые
раньше лежали прямо в HTML-файле (CRIT_OPS, TAC_OPS, KILL_OP,
UNIVERSAL_EQUIPMENT) — теперь всё в одном месте.

### 3. `tracker_logic.js`
Чистые JS-функции: создание сущностей (оператор, команда, состояние),
переходы состояния (выбрать Kill Team, изменить здоровье, использовать
снаряжение и т.д.), сериализация. Ни одна функция не создаёт и не
трогает DOM, не возвращает HTML-строку, не содержит CSS-класс или
инлайн-стиль. Каждая функция синхронная, принимает явные аргументы
(включая `gameData` из файла 2, если нужно), мутирует переданные объекты
и/или возвращает результат.

Файл написан как CommonJS-модуль (`module.exports` в конце) для удобства
тестирования в Node, но сами функции — обычный ванильный JS без
привязки к фреймворку. Их можно:
- скопировать как есть в React/Vue/Svelte-стор,
- обернуть в reducer/actions,
- использовать напрямую в vanilla JS с любым UI-слоем.

Функции проверены смоук-тестом (полный цикл: выбор команды → сборка
пула → крит-оп → архетип/tac-op → снаряжение → старт партии → игровые
действия → сериализация/десериализация) — весь сценарий проходит без
единого упоминания разметки.

## Как этим пользоваться при сборке нового UI

1. Прочитать `FUNCTIONAL_SPEC.md` — понять, что вообще нужно построить.
2. Спроектировать экраны/компоненты с нуля, ориентируясь только на
   список действий и данных из спеки — не подглядывая в старые
   скриншоты или файлы.
3. Подключить `game_data.json` как источник данных (просто заменить
   заглушку `fetch`/`import`).
4. Использовать функции из `tracker_logic.js` для всей бизнес-логики —
   не переизобретать её заново, там уже учтены все граничные случаи
   (лимиты пула, лимит снаряжения, клик по треку здоровья, сброс
   приказов при новом turning point и т.д. — см. §6 спеки).
5. Написать любой слой рендера сверху — своей формы, своих цветов,
   своей компоновки.

## Важный нюанс: `adjustPoolCount`

Функция `adjustPoolCount()` только обновляет счётчики выбора
(`team.poolCounts`) — она **не** пересобирает список операторов сама.
После каждого вызова, вернувшего `true`, нужно отдельно вызвать
`rebuildOperatorsFromKillTeam(gameData, team)`, чтобы список операторов
синхронизировался. Раздельно это сделано специально — чтобы можно было
батчить несколько изменений пула перед одной пересборкой, если это
удобнее для конкретной архитектуры UI. Подробности — в комментарии
к функции в самом файле.

## Чего здесь осознанно нет

Нет ни одной картинки, ни одного цвета, ни одного значения размера,
отступа или радиуса скругления, ни одного упоминания шрифта. Список
того, что решать заново — в конце `FUNCTIONAL_SPEC.md`, раздел 8.
