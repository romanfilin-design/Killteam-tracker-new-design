# Kill Team Solo Tracker — функциональная спецификация

Это описание ТОЛЬКО логики и данных приложения. Здесь намеренно нет ни слова
про цвета, отступы, шрифты, расположение элементов, форму карточек или любые
другие визуальные решения — это чистый contract того, что приложение должно
делать. Визуальную реализацию нужно придумать с нуля.

Приложение — трекер одной команды (kill team) для настольной игры Kill Team,
которым пользуется один игрок на своём устройстве во время партии против
другого игрока (у того — своя копия трекера).

---

## 1. Общая структура приложения

Два экрана (режима), между которыми можно переключаться в любой момент без
потери данных:

1. **Подготовка** — сборка команды и параметров партии до начала игры.
2. **Игра** — ведение партии в реальном времени: здоровье, счётчики, статусы.

Данные состояния сохраняются локально на устройстве и должны переживать
закрытие и повторное открытие приложения (персистентность). Также должен
быть экспорт всего состояния в файл и импорт обратно — чтобы можно было
сделать бэкап или перенести партию на другое устройство.

---

## 2. Источник игровых данных (внешний, редактируемый без правки кода)

Приложение обязано подгружать список доступных команд и их описание из
отдельного файла данных (не хардкодить в код), чтобы новые команды можно
было добавлять правкой одного файла. Если файл данных недоступен — должен
быть встроенный запасной набор, чтобы приложение не ломалось.

### 2.1 Структура одной команды (Kill Team)

- Название команды.
- Список архетипов, доступных этой команде (см. раздел 5).
- Состав операторов:
  - **Обязательные операторы** — всегда входят в команду. У каждого: имя,
    характеристики (см. ниже), сколько таких операторов обязательно (обычно 1).
  - **Пул на выбор** — список операторов, из которых игрок выбирает
    ограниченное количество при сборке. У каждой позиции пула:
    - имя, характеристики,
    - опциональный лимит копий одного и того же оператора (по умолчанию 1,
      у некоторых операторов может быть больше, например 2).
  - Число, которое нужно выбрать из пула суммарно (например «выбери 5 из 6»
    или «выбери 8 из 7 позиций пула, где у двух позиций лимит по 2 копии»).
- Ссылка на набор снаряжения фракции этой команды (см. 2.3), если есть.

### 2.2 Характеристики оператора

Каждый оператор (и в обязательных, и в пуле) описывается:
- **APL** — число очков действий за активацию.
- **MOVE** — дистанция передвижения (текстовое значение с дюймовым знаком,
  например `6"`).
- **SAVE** — защитный бросок (текстовое значение, например `3+`).
- **WOUNDS** — максимальное здоровье (число).
- Опционально: путь/ссылка на изображение-портрет оператора. Если не указано —
  портрета нет, приложение обязано нормально работать и без него.

### 2.3 Снаряжение

Два независимых списка:

- **Универсальное снаряжение** — общий список, доступный любой команде.
  Известные позиции (пример из уже реализованной версии, состав может
  расширяться): Ammo Cache, Razor Wire, Comms Device, Mines, Light
  Barricades, Heavy Barricade, Ladders, Portable Barricade, Utility
  Grenades, Explosive Grenades, Breaching Charge.
- **Снаряжение фракции** — своё для каждой команды (не для всех команд
  оно обязательно есть).

Каждая позиция снаряжения описывается:
- название,
- текстовое описание правила (используется как памятка в игре),
- количество доступных копий этой позиции (например «1×», «2×», либо
  «выбери 2» — когда сама позиция подразумевает выбор нескольких вариантов
  внутри себя, как у гранат),
- лимит использований за партию:
  - `null` / отсутствует — снаряжение пассивное, действует постоянно,
    использовать вручную не нужно, счётчика не требует;
  - конкретное число (1, 2, ...) — снаряжение можно «использовать» только
    это число раз за партию, нужен счётчик оставшихся использований и
    возможность отметить очередное использование;
  - особый случай «на каждый выбранный вариант» — используется для
    предметов типа гранат, где число использований равно тому, сколько
    вариантов этого предмета выбрано при подготовке.

При выборе снаряжения на экране подготовки действует **единый лимит**:
суммарно нельзя выбрать больше 4 позиций снаряжения (считая вместе
универсальное и фракционное). Каждая позиция выбирается не более одного
раза (за исключением явно указанных «выбери N» позиций, где логика выбора
внутренняя для этой позиции, а не про общий лимит 4).

### 2.4 Общие игровые операции (Crit Op)

Список из 9 позиций, единый для всех команд (не зависит от выбранной
команды). Каждая позиция:
- название,
- текст условия/действия для набора очков (мисси-экшена),
- текст условия начисления победных очков (VP).

Это выбор **одной** позиции на партию (общий для обоих игроков в реальной
игре, но в этом приложении просто хранится как выбор текущего пользователя —
игроки должны заранее договориться и выбрать одинаковую позицию каждый на
своём устройстве).

### 2.5 Kill Op

Отдельная фиксированная механика (не выбирается, а всегда доступна как
опция подсчёта):
- у команды есть «kill grade» — число от 0 до 5, растущее по мере вывода
  из строя операторов противника;
- таблица соответствия: строка — стартовое количество операторов у
  противника (диапазон значений от 5 до 14), столбцы — kill grade от 1 до
  5, значение в ячейке — сколько операторов противника нужно вывести из
  строя, чтобы достичь этого grade.
- Приложение должно позволять указать вручную, со сколькими операторами
  начал партию противник (число, разумный диапазон 1–20, по умолчанию 10),
  и по этому числу подсказывать нужную строку таблицы.
- Текущий kill grade команды — редактируемое число (кнопки увеличить/
  уменьшить), не привязано к автоматическому подсчёту (ручной трекинг).

### 2.6 Tac Op

Список операций, сгруппированных по архетипам (см. раздел 5). У каждой
позиции:
- название,
- условие раскрытия (reveal condition) — текст,
- правила/механика — текст,
- условие начисления VP — текст.

Игрок выбирает архетип своей команды, затем — одну Tac Op из списка,
доступного этому архетипу.

## 3. Состояние партии (что хранится и меняется в реальном времени)

### 3.1 Общее состояние (не привязано к конкретной команде)

- Текущий режим экрана (подготовка / игра).
- Номер текущего turning point (раунда), число, стартует с 1, можно
  увеличивать/уменьшать вручную (разумные пределы, например 1–6).
- Выбранная позиция Crit Op (ссылка на одну из 9, может быть не выбрана).
- Число операторов, с которым начал партию противник (для расчёта Kill Op,
  см. 2.5).

### 3.2 Состояние команды игрока

- Название команды (редактируемый текст, по умолчанию — название выбранного
  Kill Team или заглушка).
- Какой Kill Team выбран (ссылка на данные из раздела 2.1), может быть не
  выбран, если игрок собирает ростер вручную через импорт.
- Выбранный архетип.
- Выбранная Tac Op.
- Список операторов текущей команды (см. 3.3).
- Счётчик Command Points (CP) — число, редактируется кнопками +/-, начинает
  партию с 2.
- Счётчик Victory Points (VP) — число, редактируется кнопками +/-, стартует
  с 0.
- Текущий kill grade — число 0–5, редактируется кнопками +/-.
- Выбранное снаряжение — список позиций (из универсального и/или
  фракционного набора), не более 4 суммарно.
- Счётчик использований по каждой лимитированной позиции снаряжения
  (сколько раз уже использована).

### 3.3 Состояние одного оператора

- Уникальный идентификатор.
- Имя (редактируемый текст, по умолчанию — имя из выбранного Kill Team).
- Максимальное здоровье (число, редактируемое).
- Текущее здоровье (число, 0..максимум).
- Флаг «активировался в этот turning point» (да/нет).
- Текущий приказ: не выбран / Conceal / Engage. Циклически переключается
  по одному действию: не выбран → Conceal → Engage → Conceal → ...
- Список произвольных текстовых статус-токенов на операторе (динамический
  набор строк, не фиксированный список).
- Быстрый доступ к трём часто используемым статусам как отдельным
  переключателям: Poison, INSPIRING, Suspicion (по сути те же токены из
  списка выше, но с кнопкой быстрого включения/выключения без ввода текста).
- Опционально характеристики оператора (APL/MOVE/SAVE), если они были у
  него в данных Kill Team — используются только для отображения, не влияют
  на логику.
- Опционально путь к портрету — используется только для отображения.

## 4. Действия пользователя (что должно быть возможно сделать)

### 4.1 На экране подготовки

- Выбрать Kill Team из списка доступных (см. 2.1). При выборе состав
  операторов пересобирается согласно правилам этой команды.
- Внутри выбранного Kill Team: инкрементально добавлять/убирать операторов
  из пула по одному клику на конкретную позицию пула, с уважением к:
  - лимиту копий этой позиции (если есть),
  - общему лимиту «сколько нужно выбрать из пула»,
  - обязательные операторы всегда присутствуют и не редактируются через
    этот механизм.
  При изменении состава уже существующие операторы с тем же именем должны
  сохранять свой прогресс (здоровье, токены), а не пересоздаваться с нуля —
  важно для случая, когда игрок меняет состав после того, как уже начал
  накапливать какое-то состояние (на будущее — не критично для чистой
  подготовки, но не должно ломаться).
- Альтернативно — импортировать состав из файла (когда нет данных Kill Team
  или нужен полностью произвольный ростер): список операторов с именами и
  здоровьем. Импорт заменяет текущий автоматический состав и переводит
  команду в «ручной» режим (выбор Kill Team сбрасывается).
- Также должна быть возможность вручную полностью очистить текущий состав.
- Выбрать один общий Crit Op из списка 9 (см. 2.4). Также должна быть
  отдельная функция «случайный выбор» — при нажатии одной кнопки
  автоматически выбирается случайная позиция из 9 (равновероятно).
- Выбрать архетип команды, затем Tac Op из списка, доступного архетипу.
  Смена архетипа сбрасывает выбранную Tac Op.
- Выбрать снаряжение (универсальное и/или фракционное), с учётом общего
  лимита 4 и запрета повторного выбора одной и той же позиции.
- Переименовать команду.
- Кнопка «Начать партию» — переводит в режим «Игра». Должна быть
  недоступна (заблокирована), пока не выполнены минимальные условия:
  выбран Crit Op, у команды есть хотя бы один оператор, выбраны архетип и
  Tac Op.

### 4.2 На экране игры

- Изменять номер turning point (+/-).
- Кнопка «сброс отметок нового turning point» — единовременно у всех
  операторов команды: снимает флаг «активировался» и сбрасывает приказ
  (Conceal/Engage) обратно в «не выбран». Это логически привязано к смене
  turning point — в начале нового раунда все операторы получают приказ
  заново, поэтому старое значение приказа теряет смысл.
- Изменять CP (+/-), VP (+/-), kill grade (+/-) команды.
- Указывать вручную число операторов противника на старте (для расчёта
  Kill Op) — редактируемое число.
- Добавить нового оператора вручную (с именем-заглушкой и стандартным
  здоровьем) — на случай, если нужно докинуть кого-то не из стандартного
  состава.
- Удалить оператора (с подтверждением, т.к. необратимо).
- Для каждого оператора:
  - переименовать;
  - переключить флаг «активировался»;
  - изменить здоровье: быстрые кнопки увеличить на 1/3 и уменьшить на
    1/3/6, а также — интерактивная полоса-трек здоровья, где можно кликнуть
    по конкретному делению и здоровье сразу выставится на это значение
    (клик по уже установленному делению уменьшает на 1 относительно него —
    то есть повторный клик по текущему краю здоровья снимает одно деление);
  - редактировать максимальное здоровье вручную (число);
  - переключать приказ Conceal/Engage по клику (циклически, см. 3.3);
  - включать/выключать три быстрых статуса (Poison, INSPIRING, Suspicion)
    по одному клику каждый;
  - добавлять произвольный текстовый статус-токен (с вводом текста) и
    убирать любой конкретный токен (включая быстрые статусы — они такие
    же токены, могут быть убраны тем же способом);
  - видеть сводку по всей команде: сколько операторов активировалось из
    общего числа; сколько выведено из строя (здоровье 0) из общего числа.
- Для каждой позиции выбранного снаряжения:
  - если снаряжение пассивное — просто видно его описание, без действий;
  - если у снаряжения есть лимит использований — видно, сколько
    использований осталось из скольких всего, и доступна кнопка
    «использовать», которая уменьшает остаток на 1; когда остаток
    достигает 0, кнопка становится недоступна.
- Доступен блок-памятка «карточки этой партии» (можно свернуть/развернуть),
  который в одном месте показывает вместе, без необходимости листать
  что-то отдельно:
  - выбранный Crit Op целиком (действие + условие VP),
  - для команды: выбранная Tac Op целиком (reveal + правила + VP), список
    выбранного снаряжения,
  - таблицу Kill Op с уже подставленной нужной строкой (по числу операторов
    противника) и текущим kill grade.
- Кнопки: вернуться на экран подготовки (без потери состояния); экспорт
  всего состояния партии в файл; импорт состояния из файла (полностью
  заменяет текущее); «новая партия» — сбрасывает игровые счётчики (CP, VP,
  kill grade, здоровье до максимума, флаги активации, приказы, токены,
  использования снаряжения) до стартовых значений, но сохраняет состав
  команды, выбранные Kill Team/архетип/Tac Op/снаряжение — то есть это
  «переиграть с тем же составом», а не полная очистка.

---

## 5. Архетипы и их Tac Op (фактические данные)

Архетипы: **Recon**, **Infiltration**, **Security**, **Seek & Destroy**.
У каждого — ровно 3 позиции Tac Op в уже реализованной версии (список
может расширяться, структура должна это допускать без переделки логики).

---

## 6. Правила целостности данных (инварианты, которые логика обязана соблюдать)

- Здоровье оператора никогда не выходит за пределы [0, максимум].
- Число выбранных операторов из пула Kill Team никогда не превышает
  заявленный лимит выбора для этой команды.
- Число копий одного оператора из пула никогда не превышает его
  индивидуальный лимит копий (если задан) и по умолчанию равен 1.
- Число выбранных позиций снаряжения никогда не превышает 4.
- Использование лимитированного снаряжения никогда не превышает его лимит
  использований.
- При смене Kill Team или импорте стороннего ростера — фракционное
  снаряжение прежней команды (которое могло стать неприменимым) должно
  корректно обрабатываться (не ссылаться на несуществующие данные);
  универсальное снаряжение остаётся в любом случае, так как оно не
  привязано к конкретной команде.
- Приложение обязано корректно открывать состояние, сохранённое в
  предыдущей версии логики (то есть при появлении новых полей — заполнять
  их разумными значениями по умолчанию, а не падать с ошибкой).

---

## 7. Форматы обмена данными (для экспорта/импорта состояния и ростера)

- Экспорт состояния — полный дамп всего, что описано в разделе 3, в виде
  структурированных данных (без указания конкретного синтаксиса — важно,
  что это должно быть человекочитаемо и легко редактируемо вручную при
  необходимости, и однозначно разбираться обратно).
- Импорт ростера (частичный) — список операторов, у каждого как минимум
  имя и здоровье; опционально — название команды целиком.

---

## 8. Явно вне рамок этого документа

Ничего из следующего не является частью функциональных требований и
оставлено полностью на усмотрение новой реализации:
- как выглядят экраны, в каком порядке визуально расположены блоки;
- цвета, типографика, иконография, анимации;
- используется ли тёмная или светлая тема;
- конкретная форма кнопок, полей ввода, счётчиков;
- адаптивность под конкретные размеры экрана (кроме
  общего требования — приложением должно быть удобно пользоваться с
  телефона за игровым столом);
- как именно физически представлен блок-памятка (сворачиваемая панель,
