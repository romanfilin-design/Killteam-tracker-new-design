# Добавление новой команды — инструкция

Что автоматизировано скриптами в этой папке, а что всё равно требует
разбора руками (в сессии с Claude Code).

## Что нельзя (пока) полностью автоматизировать

Перевод правил из PDF и превращение их в структуру `game_data.json`
требует понимания контекста (что из текста правил идёт в характеристики
оператора, что — в описание снаряжения, как сформулировать краткий
русский текст для карточки-памятки). Это шаг, который делает Claude в
сессии, читая PDF и картинки.

## Важно: кэш браузера после обновления CSS/JS

`docs/index.html` подключает `css/style.css`, `js/app.js` и остальные
скрипты с query-параметром `?v=...` — это принудительно заставляет
браузер (и GitHub Pages/CDN) подтянуть новый файл вместо закэшированной
старой версии. **При любом изменении CSS/JS нужно вручную поднять это
значение** во всех трёх `<link>`/`<script>` тегах в `docs/index.html`
(например, на текущую дату) — иначе пользователи с уже открытым сайтом
могут долго видеть старую версию интерфейса, хотя `game_data.json` (без
версии в URL, подгружается через `fetch`) обновится сразу.

## Что автоматизировано

### 1. `sync_release.py` — синхронизация `docs/`

После того как `game_data.json` и/или `tracker_logic.js` в корне
проекта отредактированы (Claude делает это сам), нужно скопировать
актуальные данные/логику в `docs/` и пересобрать встроенный фолбэк.
Раньше это делалось руками через python-однострочники в чате — теперь
один скрипт:

```bash
python3 tools/sync_release.py
```

Проверяет структуру (обязательные ключи, ссылки архетипов на
`tacOpsByArchetype`, ссылки `equipment` на `factionEquipmentByTeam`) и
падает с понятной ошибкой, если что-то не так — не даст залить
битый JSON.

### 2. `smart_crop.py` — подготовка портретов (вырезание фона + кроп)

Снимает фон через **rembg** (ML-модель u2net, тот же инструмент и то же
общее окружение `Warhammer/.venv-tools/`, что использует проект
Spearhead) — работает с любым фоном фото (студийный, трава на подставке,
тени), не только с чистым белым, как было раньше. Дальше кадрирует по
альфе с небольшим отступом, вписывает в квадрат (дополняя прозрачными
полями, если пропорции не 1:1) и сохраняет как PNG с альфа-каналом.

Одиночный файл:

```bash
python3 tools/smart_crop.py "incoming_portraits/Nemesis Claw/Night Lord Warrior.jpg" \
  docs/img/portraits/nemesis-claw/warrior.png
```

Пакетно на всю папку команды:

```bash
python3 tools/smart_crop.py --batch "incoming_portraits/Nemesis Claw/" \
  docs/img/portraits/nemesis-claw/
```

Если на фото несколько миниатюр или рядом лишний предмет, который может
сбить rembg (штандарт, соседняя фигурка) — сначала вырежи область руками
по пиксельным координатам:

```bash
python3 tools/smart_crop.py "input/Night Lord Ventrilokar.jpg" \
  docs/img/portraits/nemesis-claw/ventrilokar.png \
  --manual-box 350,260,660,570
```

`--manual-box LEFT,TOP,RIGHT,BOTTOM` — пиксельные координаты
прямоугольника в исходном (некадрированном) изображении, кроп
применяется перед снятием фона. Флаг `--no-alpha` отключает вырезание
фона (просто ресайз в квадрат), если для какой-то команды оно не нужно.

Требования к исходнику — минимальные: любой формат (jpg/jpeg/png/webp),
любой фон, фигурка видна целиком и не перекрыта другими моделями/текстом.
Специально готовить квадрат в Photoshop больше не обязательно (хотя
можно — тогда просто отступов после кропа будет чуть меньше).

**Имя файла = точное имя оператора**, как оно заведено (или будет
заведено) в `game_data.json`, например `Night Lord Warrior.jpg` — по
нему идёт сопоставление файла с данными.

### Пайплайн через папку-приёмник `incoming_portraits/`

Готовые портреты кладутся в `incoming_portraits/<Название команды>/`
(см. `incoming_portraits/README.md` — там та же инструкция для быстрой
сверки). Дальше в сессии с Claude Code достаточно написать что-то вроде
«портреты для Nemesis Claw готовы, лежат в incoming_portraits» — Claude:

1. проверит файлы (разумное разрешение, имена совпадают с операторами
   команды в `game_data.json`);
2. прогонит их через `smart_crop.py --batch` в
   `docs/img/portraits/<team-slug>/`, проверит контакт-лист результата
   на предмет неудачных срезов (rembg иногда цепляет тень или соседний
   предмет — тогда пересобрать конкретный файл с `--manual-box`);
3. если у команды в `game_data.json` ещё нет ссылок `portrait` на эти
   файлы — допишет их;
4. запустит `python3 tools/sync_release.py`;
5. проверит результат в браузере;
6. очистит `incoming_portraits/<Название команды>/` (она рабочая, не
   архив, в git не коммитится).

## Полный пошаговый пайплайн добавления новой команды

1. **PDF с официальными правилами команды** кладёшь в
   `incoming_teams/<Название команды>.pdf` (см. `incoming_teams/README.md`)
   — имя файла = точное название команды. Прикладывать PDF в чат не
   нужно, я читаю его прямо по пути.
2. Если готовы и портреты, они отдельно кладутся в
   `incoming_portraits/<Название команды>/` (см. выше), с именами
   файлов = имена операторов как в PDF. Необязательно делать это
   одновременно с PDF.
3. Пишешь в сессии с Claude Code «добавь команду <Название>» (упомяни,
   если портреты тоже готовы). Дальше:
   - разбор самого PDF (постраничные картинки датакард тяжелы по
     контексту) я делегирую отдельному субагенту — он читает файл,
     переводит нужные текстовые поля на русский и возвращает готовый
     кусок JSON (характеристики операторов, состав отряда, архетипы,
     снаряжение фракции, таблицы оружия и скиллы каждого оператора —
     `weapons`/`abilities` для датакарты на игровом экране), не раздувая
     контекст текущей сессии;
   - **проверочный шаг (обязательный, не пропускать):** прежде чем
     вписывать ответ субагента в `game_data.json`, вычитываю текст
     каждой способности/правила/сноски на предмет соблюдения конвенции
     перевода — субагенты склонны мешать английские глаголы и связки
     прямо в русский текст («когда этот оператор fighting или
     retaliating» вместо «когда этот оператор сражается или отвечает
     (retaliating)»). Правило: **связки, глаголы, обычные существительные
     — переводятся; официальные игровые термины и ключевые слова
     (Range, Piercing, Lethal, APL, Move, Save, Wounds, Atk, Dmg, Hit,
     control range, killzone, turning point, INSPIRING, PSYCHIC, SUPPORT,
     Strategic Gambit, Charge, Dash, FLY и т.п.) — остаются на английском
     внутри русского текста.** Если субагент это нарушил — переписываю
     текст сам перед мёржем, не полагаясь на то, что "и так сойдёт";
   - вписываю проверенный кусок в `game_data.json` в корне проекта (по
     той же структуре, что уже используют существующие команды);
   - сопоставляю файлы из `incoming_portraits/` с операторами по именам
     и прогоняю их через `smart_crop.py --batch` в
     `docs/img/portraits/<team-slug>/` (если в официальных именах
     операторов есть диакритика вроде `Dôzr`/`Kognitâar`, а в файлах
     портретов её может не быть — сопоставляю вручную по роли, а не по
     точному совпадению строки);
   - запускаю `python3 tools/sync_release.py`, чтобы обновить
     `docs/game_data.json`, фолбэк и `tracker_logic.js` в `docs/`;
   - проверяю результат в браузере (preview): выбор команды, лимиты
     пула, старт партии, портреты, датакарты операторов, правило
     фракции на игровом экране.
4. **Ты просматриваешь `docs/`** (при желании — открываешь
   `docs/index.html` через локальный сервер) и, если всё ок, делаешь
   `git add` / `git commit` / `git push` сам (или просишь меня «залей
   на гит»).

Шаг 3 можно запросить без пересказа этого списка — достаточно положить
файлы по нужным путям и написать «добавь команду».
