# Стартовый бриф — дальнейшая разработка трекера

Вставь этот файл целиком первым сообщением в новую сессию Claude Code,
открытую в папке `Killteam new design`. Это ветка для развития функций
самого трекера (не для добавления новых команд — для этого есть
`SESSION_START_NEW_TEAMS.md`).

## Что это за проект

Kill Team Solo Tracker — статический сайт для соло-игры в Warhammer
40,000: Kill Team. Прод:
https://romanfilin-design.github.io/Killteam-tracker-new-design/
Без бэкенда, без build-шага, состояние партии живёт в `localStorage`.

## Архитектура

- **`game_data.json`** (корень) — канонические данные всех команд:
  архетипы, пул операторов, снаряжение, faction rules, лор,
  `factionChoices`, `statusTokens`. Редактируется только в корне.
- **`tracker_logic.js`** (корень) — чистая игровая логика без DOM:
  создание команды, старт партии, `factionChoices`/`statusTokens`
  хелперы, импорт/экспорт состояния. Тоже редактируется только в корне.
- **`tools/sync_release.py`** копирует оба файла из корня в `docs/`,
  собирает встроенный JS-фолбэк на случай отказа `fetch` данных, и
  валидирует схему (упадёт с понятной ошибкой на битых ссылках). Запускай
  после любого изменения `game_data.json`/`tracker_logic.js` — **никогда
  не редактируй копии внутри `docs/` напрямую**, следующий sync их
  перезапишет.
- **`docs/js/app.js`** — единственный слой рендера/UI. Ванильный JS
  IIFE, глобальный `ACTIONS`-объект как event-роутер для всех кликов
  (data-action атрибуты в разметке). У этого файла нет отдельного
  «исходника» — правки вносятся прямо в `docs/js/app.js`.
- **`docs/css/style.css`** — текущий дизайн: industrial brutalism
  (тёмный, резкие углы, моно-шрифт для чисел). Компактная сетка
  `.operator__stats` (3 колонки характеристик под именем), мобильный
  скролл для `.pick-list`. Есть `--info`/`--info-bright` цветовые
  переменные для нейтральных/информационных акцентов (Conceal — синий,
  Activated — нейтральный, не «опасный» красный).
- **`docs/index.html`** — три `<link>/<script>` тега с `?v=YYYYMMDDx`
  кэш-бастером. **Поднимай версию на любое изменение CSS/JS**, иначе
  браузер пользователя может закэшировать старую версию произвольно
  долго.
- Деплой — `.github/workflows/pages.yml`, GitHub Actions
  (`upload-pages-artifact` + `deploy-pages`), триггер — push в `main`.
  Раньше был легаси Jekyll-пайплайн («Deploy from a branch»), который
  зависал на неопределённое время — заменён этим workflow-based
  подходом; текущий деплой занимает ~15-20 секунд.

## Механизм factionChoices / statusTokens (введён в этой сессии)

Обобщённый механизм для командных (не операторских) особенностей,
которые не были нужны первым 5 командам, но появились у Angels of
Death (Chapter Tactics + Combat Doctrine) и рассчитаны на любую будущую
команду с похожей механикой.

- **`factionChoices`** — «выбери N из списка» на уровне команды.
  - `scope: 'setup'` — выбор фиксируется один раз при сборке ростера
    (Chapter Tactics). Рендерится `renderFactionChoicesSetupPanel()`.
  - `scope: 'game'` — можно переключать по ходу партии (Combat
    Doctrine — Strategy Ploy). Рендерится
    `renderFactionChoicesGamePanel()`, с текстом-подсказкой (hint) по
    каждому варианту.
  - Логика в `tracker_logic.js`: `findFactionChoiceDef`,
    `setFactionChoiceSlot`, `areFactionChoicesComplete`. Последняя
    участвует в `isReadyToStart()` — нельзя начать партию с незакрытым
    setup-выбором.
  - `renderFactionRulePanel()` фильтрует из общего списка faction rules
    те, что уже показаны интерактивным `factionChoices`-селектором —
    чтобы правило не дублировалось текстом и виджетом одновременно.
  - Кастомные faction-choices команды также попадают в
    «Карточки этой партии» (cheat-sheet) через `factionChoicesHtml` в
    `renderCheatSheet()` — это generic-код, работает для любой команды,
    ничего командо-специфичного туда не зашито.
- **`statusTokens`** — быстрые чипы-статусы на карточке оператора.
  - `target: 'friendly'` — рендерятся на своей карточке, доступны через
    `friendlyStatusTokenNames(gameData, killTeamName)`.
  - `target: 'enemy'` (Poison у Death Guard, Grudge у Hearthkyn) —
    **данные хранятся, но не рендерятся нигде сейчас**. Это осознанный
    задел под следующую фичу (см. ниже) — не удаляй и не «доделывай
    просто так», это ждёт своей вкладки «Противник».
- При смене команды (`ACTIONS.selectKillTeam`) сбрасываются: `archetype`,
  `tacOpId`, `equipmentIds`, `equipmentUsed`, `factionChoices`,
  `critOpId` — чтобы старые выборы не блокировали проверку готовности к
  игре под новой командой.

## Известные грабли этой сессии (не наступай снова)

- **`preview_console_logs` не ловит необработанные исключения** —
  только явные `console.*` вызовы. Проверено намеренным `throw`. Если
  подозреваешь краш — смотри `preview_screenshot`, DOM через
  `preview_snapshot`/`preview_eval`, и содержимое
  `localStorage.getItem('kt_solo_tracker_state_v1')` напрямую. Также
  этот тул иногда показывает залипшие старые записи после нескольких
  reload — не единственный источник правды.
- **Старое состояние в localStorage может не иметь новых полей** —
  при любом изменении схемы состояния партии добавляй backfill в
  `importState()` (пример: `if (!t.factionChoices) t.factionChoices = {}`),
  иначе у пользователей с уже сохранённой партией будет краш при загрузке.
- **Материалы от субагентов требуют вычитки**: даже когда субагент
  делает только код (не перевод/данные), перепроверяй итог напрямую —
  в этой сессии субагенты плодили `&amp;`-контаминацию, забытые
  `maxCopies`, мета-комментарии в player-facing тексте.
- **Git push только по явному разрешению** («деплой»/«залей»/«заливай»
  в том же сообщении) — не пуш по инерции даже если предыдущий пуш был
  только что одобрен.
- `gh` CLI авторизован через fine-grained PAT в keychain (не env var) —
  переживает разные сессии/процессы. Если пользователь просит проверить
  Pages/Actions статус — используй `gh api repos/romanfilin-design/Killteam-tracker-new-design/pages`
  и `gh run list`.

## Активное направление (обсуждалось, не согласовано к реализации)

**Комната синхронизации для двух игроков** — следующая крупная фича,
пока только обсуждение архитектуры, кода не писалось:
- Общий Crit Op (рандомный, с возможностью переброса до начала игры).
- Вкладка «Противник»: карточка оператора врага без кнопок +/- HP, но
  с баром приказов, активацией, и `statusTokens` с `target:'enemy'`
  (эти данные уже подготовлены в схеме команд, ждут UI).
- Рекомендация по реализации: Firebase/Firestore с парой по коду
  комнаты (проще и надёжнее WebRTC P2P для двух непостоянно
  подключённых игроков, укладывается в бесплатный тир Firebase при
  таком масштабе использования). Нужен экран входа/создания комнаты.
- Не начинай реализацию без явного запроса — это стратегическое
  решение, которое пользователь ещё не подтвердил.
