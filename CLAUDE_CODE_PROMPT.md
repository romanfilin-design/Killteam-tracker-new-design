# Задача: собрать Kill Team Solo Tracker с нуля, визуальный стиль — grimdark sci-fi fantasy

## Контекст

В папке проекта лежат три файла — это полная функциональная спецификация
приложения, без единого дизайнерского решения:

- `FUNCTIONAL_SPEC.md` — что приложение должно делать: экраны, данные,
  действия пользователя, правила целостности. Читай первым.
- `game_data.json` — фактические игровые данные (правила настольной игры,
  список команд, снаряжения, миссий). Подключается как источник данных.
- `tracker_logic.js` — готовый, протестированный слой бизнес-логики
  (создание сущностей, переходы состояния, сериализация). Не содержит
  ни HTML, ни CSS. Используй эти функции как есть или адаптируй в свою
  архитектуру (стор/reducer/composable) — не переписывай логику с нуля,
  там уже учтены граничные случаи (лимиты выбора, сброс приказов при
  новом раунде, клик по треку здоровья и т.д.).

Прочитай все три файла перед тем, как начать. `README.md` рядом объясняет,
как они связаны.

## Что нужно построить

Мобильный веб-интерфейс (в первую очередь — использование с телефона за
игровым столом) поверх этой логики. Один HTML/CSS/JS проект (или React —
на твоё усмотрение по стеку, но без сложного build-пайплайна: результат
должен быть легко разместить на статическом хостинге типа GitHub Pages).
Данные сохраняются в `localStorage` браузера.

Реализуй оба экрана и все действия пользователя, перечисленные в
`FUNCTIONAL_SPEC.md`, разделы 4.1 и 4.2, полностью — это не референс для
частичной реализации, а обязательный список функциональности.

## Визуальное направление: grimdark sci-fi fantasy

Это главная часть задачи — предыдущая попытка визуально слишком копировала
референсный источник, и от этого нужно уйти. Придумай дизайн-язык с нуля,
не оглядываясь на то, как выглядят существующие карточки/трекеры для этой
игры или подобных. Никаких скриншотов-референсов не прикладываю намеренно —
хочу самостоятельную интерпретацию настроения, а не копию чего-либо
конкретного.

Настроение, которое должен передавать интерфейс — мрачное далёкое будущее
на грани фэнтези: холодный индустриальный ужас, обветшалая, но всё ещё
функционирующая военная техника, ощущение, что интерфейс сделан внутри
вселенной (диегетический UI) — как будто это терминал, которым реально
пользуется оператор в поле, а не приложение с App Store.

Отправные точки для настроения (интерпретируй свободно, не как чек-лист
элементов, а как вайб):
- тяжёлая, промышленная типографика — шрифты, которые выглядят
  трафаретными/выштампованными на металле, а не гладким UI-шрифтом;
- ощущение изношенного металла, потёртостей, лёгкого шума/зерна текстуры
  вместо идеально гладких плоских поверхностей;
- контраст тьмы и точечного тревожного света — не плоская тёмная тема
  «просто чёрный фон», а ощущение освещения одним источником в темноте;
- острые, рубленые формы вместо мягких скруглений — фаски, срезанные
  углы, а не привычные скруглённые карточки типичного мобильного приложения;
- сдержанная, немногословная палитра: тёмная основа + один-два акцентных
  цвета высокой тревожности (кроваво-ржавый, тлеющий оранжевый,
  предупреждающий жёлтый, холодный аварийный синий — выбери что-то одно
  как доминанту, не смешивай все сразу);
- ощущение статусной панели боевой техники — индикаторы здоровья,
  показатели готовности выглядят как приборная панель, а не как progress
  bar из дизайн-системы;
- никакой мимикрии под iOS/Material Design — интерфейс не должен читаться
  как «ещё одно приложение», должен ощущаться как часть вымышленного мира.

Явно избегай:
- прямого воспроизведения фирменного стиля, логотипов, шрифтов конкретной
  существующей настольной игры или медиафраншизы — нужен собственный,
  вдохновлённый общим настроением жанра «grimdark sci-fi», но не
  копирующий чужую айдентику дизайн;
- плоского «обычного тёмного режима» (тёмно-серый фон + акцентный цвет
  как у типичного SaaS-дэшборда) — это не то настроение;
- избыточной сложности, из-за которой интерфейс станет неудобно
  использовать за реальным игровым столом одной рукой на телефоне: стиль
  не должен мешать быстро находить кнопки и читать числа во время игры.

## Приоритет: функциональность превыше эффектности

Это инструмент для использования во время реальной игры, счёт идёт на
секунды между действиями. Все интерактивные элементы (кнопки +/-, счётчики
здоровья, переключатели статусов) должны быть крупными, однозначно
кликабельными, с очевидной обратной связью на нажатие. Стилизация не
должна снижать читаемость чисел (здоровье, CP, VP) — это самое частое,
что игрок будет считывать взглядом за секунду.

## Технические ограничения

- Мобильный экран — основной сценарий использования, но интерфейс не
  должен ломаться и на широком экране (десктоп/планшет) — см. в спеке
  требование об адаптивности.
- Персистентность через `localStorage`, без бэкенда.
- Загрузка `game_data.json` — через относительный путь (`fetch('./game_data.json')`
  или эквивалент в выбранном стеке), с обработкой ситуации, когда файл
  недоступен (см. `FUNCTIONAL_SPEC.md`, раздел 2, про обязательный
  запасной набор данных).
- Не используй внешние платные шрифты/иконки, требующие API-ключей;
  подключаемые через CDN бесплатные шрифты (Google Fonts и подобные) —
  нормально.

## Definition of done

- Реализованы оба экрана и все пункты действий пользователя из
  `FUNCTIONAL_SPEC.md` §4.1–4.2 без исключений.
- Приложение проходит полный ручной сценарий: выбрать команду → набрать
  состав → выбрать Crit Op (включая случайный выбор) → выбрать архетип и
  Tac Op → выбрать снаряжение → начать партию → менять здоровье/приказы/
  статусы у операторов → использовать лимитированное снаряжение → сменить
  turning point со сбросом отметок → сохранить и перезагрузить страницу
  без потери прогресса → экспортировать/импортировать состояние → начать
  новую партию с тем же составом.
- Полученный визуальный результат не похож на типичный светлый/тёмный
  UI мобильного приложения общего назначения и не похож на официальные
  материалы конкретной настольной игры — это самостоятельная стилизация
  под настроение, описанное выше.
